DAY 11非阻塞 I/O进入事件驱动前的地基

今日目标：把 Day 10 的连接处理改成非阻塞 fd，能正确处理 EAGAIN、短读、短写和半关闭。

核心概念

  • O_NONBLOCK 改变的是系统调用返回方式，不改变 TCP 字节流语义。
  • EAGAIN/EWOULDBLOCK 表示当前不能继续，不是连接出错。
  • 非阻塞写必须维护输出缓冲区，不能假设一次 send 成功。
  • 连接对象需要保存读缓冲、写缓冲、解析状态和关闭状态。

命令与代码

fcntl(fd, F_SETFL, old_flags | O_NONBLOCK);
ss -antp | grep 8080
strace -e trace=recvfrom,sendto ./server

编码任务

  1. 封装 set_nonblock(fd)。
  2. 把连接 fd 设为非阻塞，循环读到 EAGAIN 为止。
  3. 实现连接级输出缓冲，短写后保留剩余数据。
  4. 写一份状态说明：什么时候关闭连接，什么时候继续等待。

验收

危险信号

如果代码里出现“send 失败就 close”的粗暴逻辑，说明还没真正进入非阻塞模型。

DAY 12select / poll先理解多路复用本质

今日目标：用 poll 写一个单线程多客户端 Echo Server，理解“一个线程管理多个 fd”的基本方式。

理解重点

  • I/O 多路复用不是并行执行，而是等待多个 fd 的就绪事件。
  • select 有 fd 集合和数量限制，poll 用数组表达 fd 集合。
  • 就绪只是“现在调用大概率不会阻塞”，不等于读写一定完成。
  • 仍然需要非阻塞和循环读写。

实验

poll(fds.data(), fds.size(), timeout_ms);
revents & POLLIN
revents & POLLOUT
revents & POLLHUP

编码任务

  1. 用 poll 管理监听 fd 和所有连接 fd。
  2. 客户端发来长度前缀消息后回显。
  3. 只有连接有待发送数据时才关注 POLLOUT。
  4. 连接关闭后从数组中移除，避免无效 fd 残留。

验收

复盘题

为什么 poll 每次都要线性扫描 fd 数组？连接数量很大时，这会带来什么成本？

DAY 13epoll LTLinux 高并发 I/O 入口

今日目标：用 epoll 的 LT 模式重写 Day 12 服务端，掌握 add/mod/del 和事件循环。

epoll_create1
→
epoll_ctl ADD
→
epoll_wait
→
处理事件
→
MOD / DEL

理解重点

  • epoll 管理的是内核中的兴趣列表和就绪列表。
  • LT 模式下，只要条件仍满足，事件会再次被通知。
  • EPOLLIN、EPOLLOUT、EPOLLERR、EPOLLHUP 要一起考虑。
  • data.ptr 可以指向连接对象，避免全局 fd 查找。

骨架

int ep = epoll_create1(EPOLL_CLOEXEC);
epoll_ctl(ep, EPOLL_CTL_ADD, listen_fd, &ev);
while (running) {
  int n = epoll_wait(ep, events, max_events, timeout);
  for (int i = 0; i < n; ++i) handle(events[i]);
}

编码任务

  1. 把 poll 版本迁移到 epoll LT。
  2. 监听 fd 可读时循环 accept 到 EAGAIN。
  3. 连接可读时读到 EAGAIN；可写时写到 EAGAIN 或清空缓冲。
  4. 输出缓冲为空后取消关注 EPOLLOUT。

验收

观察命令

strace -e trace=epoll_wait,epoll_ctl ./server
ss -antp | grep 8080
ls /proc/<PID>/fd | wc -l
DAY 14epoll ET + 定时器边沿触发与连接超时

今日目标：理解 ET 模式为什么必须配合非阻塞和“读写到 EAGAIN”，并实现空闲连接超时清理。

核心概念

  • ET 是状态变化时通知；没有读干净，后续可能不再通知。
  • ET 写法的关键不是加 EPOLLET，而是事件处理循环必须完整。
  • 定时器用于清理空闲连接、心跳超时和周期任务。
  • 可以先用最小堆定时器，不急着上时间轮。

关键循环

for (;;) {
  ssize_t n = recv(fd, buf, sizeof(buf), 0);
  if (n > 0) append(buf, n);
  else if (n == 0) close_conn();
  else if (errno == EAGAIN) break;
  else if (errno == EINTR) continue;
  else close_conn();
}

编码任务

  1. 为 epoll 事件加上 EPOLLET，通过测试证明无消息丢失。
  2. 实现基于 steady_clock 的连接最近活跃时间。
  3. 每次 epoll_wait 使用最近定时器计算 timeout。
  4. 超过 60 秒无读写的连接自动关闭，并输出日志。

验收

口述题

LT 和 ET 的区别，不要只说“水平触发”和“边沿触发”，要能结合缓冲区状态解释。

DAY 15Reactor 模型从循环到架构

今日目标：把 epoll 服务端重构为 Reactor：事件循环、Channel、Acceptor、Connection 边界清晰。

EventLoop
→
Poller / Epoll
→
Channel
→
Callback
→
Connection

对象边界

  • EventLoop：拥有 epoll fd，负责循环和任务执行。
  • Channel：封装 fd 关心的事件和回调。
  • Acceptor：只负责监听和新连接创建。
  • Connection：拥有读写缓冲、协议解析和关闭流程。

不要过度设计

今天先做单 Reactor 单线程。不要同时引入多 Reactor、线程池、协程和复杂模板，否则很难定位问题。

重构任务

  1. 把散落的 epoll_ctl 调用收敛到 Poller。
  2. 每个 fd 对应一个 Channel，事件到达后调用回调。
  3. Connection 提供 send、shutdown、close 接口。
  4. 保持原有 Echo 和长度前缀协议测试全部通过。

验收

复盘题

Reactor 解决的是代码组织和事件分发问题，不会自动让业务变快。它的收益和代价分别是什么？

DAY 16业务分发与线程池Reactor 不被慢任务拖死

今日目标：让 I/O 线程只做网络事件，耗时业务丢给工作线程，并安全地把结果投递回 EventLoop。

核心概念

  • I/O 线程不能执行慢 SQL、复杂计算或长时间阻塞操作。
  • 跨线程操作连接对象需要明确线程归属。
  • 常见做法：业务线程产出结果，投递回对应 EventLoop 再发送。
  • eventfd 可用于唤醒事件循环。

任务模型

onMessage(conn, msg):
  workerPool.submit([connId, msg] {
    result = handleBusiness(msg);
    loop.runInLoop([connId, result] {
      conn.send(result);
    });
  });

编码任务

  1. 把 Day 10 线程池接到 Reactor 后面。
  2. 实现 EventLoop::runInLoop 或 queueInLoop。
  3. 用 eventfd 唤醒阻塞在 epoll_wait 的 I/O 线程。
  4. 连接关闭后，投递回来的旧任务不能访问已释放对象。

验收

工程判断

这一阶段先用 connId 或弱引用规避悬空访问。不要为了“优雅”引入难以解释的共享所有权。

DAY 18MySQL 连接池避免业务线程反复建连

今日目标：实现一个简单可靠的 MySQL 连接池，并把数据库访问接入业务线程池。

理解重点

  • 数据库连接是昂贵资源，不能每条消息都新建连接。
  • 连接池需要最大连接数、等待队列、超时、健康检查。
  • 连接借出后必须归还，RAII guard 很适合做这件事。
  • 事务要明确提交和回滚边界。

接口草图

class MysqlPool {
 public:
  MysqlConnGuard acquire(Duration timeout);
};

class MysqlConnGuard {
 public:
  ~MysqlConnGuard(); // return to pool
};

编码任务

  1. 连接池启动时创建最小连接数。
  2. 业务线程从池里借连接，超时则返回“系统繁忙”。
  3. 模拟连接断开，下一次借出前能检测并重连。
  4. 注册用户和加入房间放进事务，失败时回滚。

验收

复盘题

连接池满了应该排队、失败还是降级？不同选择对延迟和稳定性有什么影响？

DAY 19Redis 基础接入在线状态与会话

今日目标：用 Redis 保存在线用户、登录 token 和房间成员缓存，理解缓存不是数据库的替代品。

使用边界

  • MySQL 保存权威数据，Redis 保存快取、状态、限流和临时数据。
  • 在线状态适合 Redis，因为它随连接变化，不需要长期持久化。
  • token 需要过期时间，避免永久有效。
  • 缓存命中失败必须能回源 MySQL 或返回明确错误。

Key 设计

user:token:<token> -> user_id  EX 86400
user:online:<user_id> -> conn_id EX 120
room:members:<room_id> -> set(user_id)
rate:login:<ip> -> counter EX 60

编码任务

  1. 登录成功后生成 token，并写入 Redis。
  2. 连接认证时用 token 找 user_id。
  3. 连接建立写在线状态，断开时删除或等待过期。
  4. 缓存房间成员列表，并提供失效策略。

验收

命令观察

redis-cli KEYS 'user:*'
redis-cli TTL user:token:xxx
redis-cli SMEMBERS room:members:1
DAY 20Redis 限流与 Pub/Sub保护服务与跨进程广播

今日目标：实现登录限流和基础 Pub/Sub，为后续多进程聊天室打下边界。

核心概念

  • 限流不是安全全部，但可以降低暴力尝试和误用流量。
  • 计数器限流简单，滑动窗口更准确，令牌桶更平滑。
  • Redis Pub/Sub 是实时广播，不保证离线可靠投递。
  • 可靠离线消息仍应落 MySQL。

限流草图

INCR rate:login:<ip>
EXPIRE rate:login:<ip> 60
if count > 20:
  reject login

编码任务

  1. 给登录接口加 IP 维度限流。
  2. 给发送消息加用户维度限流。
  3. 实现 room:<id> 频道发布和订阅。
  4. 明确 Pub/Sub 断线期间消息如何处理：在线走推送，历史从 MySQL 拉。

验收

复盘题

为什么 Pub/Sub 不能当消息队列？如果要可靠队列，应考虑 Redis Streams、MQ 或数据库表。
