DAY 01Linux 开发环境终端、权限、编译、Git

今日目标：在 Linux 中独立创建、编译、运行和提交一个 C++ 项目，不依赖 IDE 的运行按钮。

理解清单 · 50 min

  • 目录树、绝对路径与相对路径
  • 普通文件、目录、隐藏文件
  • 用户/组与 rwx 权限
  • 编译、链接、可执行文件的关系

命令实验 · 45 min

pwd; ls -lah
mkdir -p linux-server/labs/day01
cd linux-server
touch README.md labs/day01/hello.cpp
chmod u+x your_script.sh
git init && git status

编码任务 · 85 min

  1. 写一个输出编译时间、进程 ID 的 hello.cpp。
  2. 分别执行预处理、编译、汇编、链接阶段，观察中间文件。
  3. 创建最小 CMakeLists.txt，用 out-of-source build 构建。
  4. 故意制造语法错误、链接错误、无执行权限，记录报错差异。

建议命令

g++ -E hello.cpp -o hello.i
g++ -S hello.i -o hello.s
g++ -c hello.s -o hello.o
g++ hello.o -o hello
file hello
ldd hello

DAY 02进程与信号看见程序如何活着

今日目标：能找到一个运行中的程序，观察它的父子关系、资源状态，并选择合适信号结束它。

核心概念 · 55 min

  • 程序是文件，进程是一次运行实例
  • PID、PPID、进程状态 R/S/D/T/Z
  • 前台、后台、作业控制
  • SIGTERM 与 SIGKILL 的差异
  • /proc 是内核暴露的运行时视图

观察命令 · 50 min

./loop &
jobs -l
ps -ef --forest
top -p <PID>
cat /proc/<PID>/status
ls -l /proc/<PID>/fd
kill -TERM <PID>

编码与故障实验 · 95 min

  1. 写循环程序，每秒打印 PID，并注册 SIGINT/SIGTERM 处理函数。
  2. 先用 Ctrl+C，再用 kill -TERM，观察清理逻辑是否执行。
  3. 用 fork() 创建子进程，打印 PID/PPID；父进程暂不 wait，短暂观察僵尸状态后修复。
  4. 记录：为什么线上服务通常先发 TERM，而不是直接 KILL？

口述题

进程与程序有什么区别？僵尸进程还占多少资源？孤儿进程会怎样？为什么 SIGKILL 无法捕获？

理解重点 · 70 min

  • IP 负责寻址，端口定位进程
  • 网络字节序、IPv4 与回环地址
  • 序列号、确认号、可靠有序字节流
  • 握手为什么需要三次
  • 主动关闭方为什么进入 TIME_WAIT

抓包实验 · 90 min

sudo tcpdump -i any -nn 'tcp port 8080'
# 另一终端：
nc -l 8080
# 再一终端：
nc 127.0.0.1 8080

必须记录的证据

  1. 截图或抄录握手三个包的 flags、seq、ack。
  2. 分别让客户端和服务端主动关闭，观察状态变化。
  3. 用自己的话回答：如果只有两次握手，旧 SYN 为什么可能制造错误连接？
  4. 区分 TIME_WAIT 与 CLOSE_WAIT：前者通常是协议状态，后者常暴露应用未关闭连接。

DAY 06Client + Echo建立完整通信闭环

今日目标：完成自己的 TCP 客户端与可重复收发的 Echo Server，正确识别 EOF 和错误。

关键语义 · 55 min

  • connect 触发握手
  • recv == 0 表示对端有序关闭写方向
  • recv < 0 才是错误，且需检查 errno
  • send 成功不代表对端业务已经处理
  • 一次 send 与一次 recv 没有对应关系

编码任务 · 110 min

  1. 客户端读取 stdin，发送到服务端并打印回显。
  2. 服务端循环 recv，再循环 send 回全部数据。
  3. 客户端输入 /quit 时执行 shutdown。
  4. 封装 send_all，处理短写和 EINTR。

故障矩阵 · 40 min

  • 先启动客户端，服务端未监听
  • 通信中杀掉服务端
  • 客户端直接关闭终端
  • 发送空行和超过缓冲区的大字符串

验收

今天最重要的一句话：TCP 提供可靠、有序的字节流，但不替应用保存“消息边界”。这会直接引出 Day 7。
DAY 07消息协议解决粘包与半包

今日目标：设计并实现“4 字节长度头 + 消息体”协议，无论数据怎样拆分到达都能正确解析。

理解重点 · 55 min

  • “粘包”不是 TCP 出错，而是应用错误假设边界
  • 定长、分隔符、长度前缀三类方案
  • 长度字段需要网络字节序
  • 必须限制最大消息长度，防止内存攻击

编码任务 · 110 min

  1. 实现 encode(message)。
  2. 实现可累积字节的接收 Buffer。
  3. 循环判断是否够 4 字节头、是否够完整 body。
  4. 解析后从 Buffer 消费该帧，继续解析下一帧。

必须通过的 4 个测试

[完整] header + body
[半包] 2-byte header | rest
[粘包] frame1 + frame2
[混合] frame1 + half(frame2)

拒绝：length > 1 MiB

DAY 08RAII 与所有权让资源生命周期可靠

今日目标：使用现代 C++ 明确 fd 的唯一所有权，让任何返回路径和异常路径都不会泄漏资源。

核心概念 · 70 min

  • 栈对象生命周期与确定性析构
  • RAII：构造获得资源，析构释放资源
  • 禁用复制、支持移动
  • unique_ptr 默认优先
  • shared_ptr 的共享所有权成本与环

Fd 封装接口

class UniqueFd {
 public:
  explicit UniqueFd(int fd = -1);
  ~UniqueFd();
  UniqueFd(const UniqueFd&) = delete;
  UniqueFd& operator=(const UniqueFd&) = delete;
  UniqueFd(UniqueFd&&) noexcept;
  UniqueFd& operator=(UniqueFd&&) noexcept;
  int get() const;
};

重构任务 · 95 min

  1. 实现 UniqueFd，移动后源对象必须变成无效状态。
  2. 用它重构 Day 5-7 的客户端、服务端和错误路径。
  3. 用 lsof 或 /proc/PID/fd 比较重构前后。
  4. 故意在 accept 后提前 return，确认 fd 仍被关闭。

工具检查

g++ -fsanitize=address,undefined \
    -fno-omit-frame-pointer -g ...
ASAN_OPTIONS=detect_leaks=1 ./server
valgrind --leak-check=full ./server

理解重点 · 65 min

  • 线程共享地址空间，但有独立栈
  • 数据竞争意味着未定义行为
  • 临界区、互斥与锁粒度

  • 始终用谓词循环处理虚假唤醒

实验顺序 · 105 min

  1. 两个线程各自执行 100 万次 counter++。
  2. 多跑几次观察错误结果，用 TSan 确认证据。
  3. 用 lock_guard 修复并理解代价。
  4. 实现带 push/pop/close 的阻塞队列。

关键模式

std::unique_lock lock(mutex_);
cv_.wait(lock, [this] {
  return closed_ || !queue_.empty();
});
if (closed_ && queue_.empty()) return false;
// pop task while holding the lock

验收

暂不做：无锁队列和复杂内存序。先把互斥、条件等待、生命周期和退出路径做正确，收益远高于追逐高级术语。
